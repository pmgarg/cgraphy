# Cgraphy — Design Spec

**Date:** 2026-07-08 (revised same day: renamed graphy → cgraphy; added
research-grade extensions)
**Status:** Approved by user (brainstorming session)

## Summary

Cgraphy is an open-source tool that converts a codebase into a knowledge graph
(nodes = files, classes, functions; edges = calls, imports, inherits) and serves
it to AI coding assistants over MCP (Model Context Protocol). Instead of an
agent re-reading full files on every prompt, it queries the graph and receives a
compact, relevant subgraph — reducing token usage and improving answers on
short or vague prompts. A minimal built-in viewer lets humans explore the same
graph visually.

## Goals

1. **LLM context efficiency (primary):** an agent can orient itself in a repo
   from ~1–2K tokens of graph output instead of reading dozens of files.
2. **Works everywhere:** one MCP server usable from Claude Code, Codex CLI,
   Gemini CLI, Cursor, and VSCode — configured with one line each.
3. **Any language:** structural extraction for any language with a tree-sitter
   grammar; full-fidelity extraction for the major ones.
4. **Zero configuration:** no API keys, no databases, no services. One command
   to index, one config line to serve.
5. **Human visualization (secondary):** a minimal local graph viewer.

## Non-Goals (v1)

- Embedding/vector search (schema leaves room; explicit v2 path).
- Team-shared or hosted graphs; committable graph export (possible v2
  `cgraphy export`).
- Editor extension UI; rich visualization app.
- Windows-specific packaging work beyond what PyPI wheels give for free.

## Architecture

Single Python package `cgraphy` (Python ≥ 3.10), installable via
`pip install cgraphy` or run via `uvx cgraphy`. Three subcommands over one core
library:

```
cgraphy index [path]     # build/update the graph
cgraphy serve            # MCP server over stdio
cgraphy view [--port]    # local HTML graph viewer
```

State lives in a single SQLite file at `.cgraphy/graph.db` inside the indexed
repo (gitignored by default; `cgraphy index` appends `.cgraphy/` to `.gitignore`
if the repo has one).

### Storage: SQLite

At codebase scale (thousands of nodes), recursive SQL traverses the graph in
milliseconds. A single dependency-free file beats a graph database for an
open-source developer tool.

**Tables**

- `nodes`: `id, kind, name, qualified_name, language, file_path, line_start,
  line_end, signature, summary, summary_hash, parse_error, rank`
  - kinds: `file`, `module`, `class`, `function`, `method`, `variable`
  - `rank`: PageRank importance score computed over the edge graph after each
    index (see Research-Grade Features).
- `edges`: `source_id, target_id, kind, weight`
  - kinds: `contains`, `imports`, `calls`, `inherits`, `references`,
    `co_changes`
  - `weight`: 1.0 for structural edges; co-change strength for `co_changes`.
- `files`: `path, content_hash, last_indexed` — drives incremental re-indexing.
- FTS5 virtual table over `name + qualified_name + summary` for search.

## Language Support (pluggable, two tiers)

Parsing uses `tree-sitter-language-pack` (prebuilt grammars for 100+
languages; no user-side compilation).

- **Tier 1 — full fidelity:** Python, TypeScript/JavaScript, Java, C, C++, Go,
  Rust. Hand-written `.scm` query files + small per-language resolvers extract
  definitions, imports/includes, call edges, and inheritance. Adding a Tier-1
  language later = adding a query file + resolver (contribution-friendly).
- **Tier 2 — generic fallback:** any other tree-sitter language gets definition
  nodes (functions/classes/types via generic queries) and file-level nodes with
  LLM summaries. Searchable and summarized, without precise call edges.
- **Non-code files** (YAML, JSON, Markdown, SQL, …) get `file` nodes with LLM
  summaries so config and docs participate in retrieval.

## Indexing Pipeline

1. Walk the repo respecting `.gitignore` plus optional `.cgraphyignore`.
2. For each new/changed file (content hash): detect language → tree-sitter
   parse → Tier-1 or generic queries → emit nodes/edges. Unchanged files are
   skipped (incremental re-index is milliseconds).
3. Second pass: cross-file edge resolution by qualified name, best-effort.
   Unresolved calls are stored as `references` to a name, never dropped.

## LLM Summarization (no API key required)

Summaries are generated by the **host agent itself** via MCP tools, so Cgraphy
needs no LLM credentials and works identically in every assistant:

1. After indexing, nodes have structure but empty `summary`.
2. `cgraphy_enrich()` returns a batch of unsummarized symbols with source
   snippets; the agent responds via `cgraphy_store_summaries(...)` with a
   one-line summary per symbol, looping until none remain (the tool
   description instructs this).
3. Summaries are keyed by `summary_hash` of the source text — editing a
   function invalidates only its own summary.
4. Optional escape hatch: `cgraphy index --summarize --api-key ...` to pre-bake
   summaries (e.g., in CI) using the user's own key.

## MCP Tool Surface (5 tools, stdio transport)

- `cgraphy_overview()` — repo map in ~1–2K tokens: top-level modules with
  summaries and the highest-PageRank entry points. The replacement for "read
  20 files to orient."
- `cgraphy_search(query)` — FTS5 match over names + summaries, re-ranked by
  a blend of FTS score and PageRank; returns nodes with `file:line` and
  summaries.
- `cgraphy_context(symbol_or_file, token_budget=2000)` — budget-constrained
  relevant subgraph: greedy expansion from the target node outward, ordered by
  edge weight × neighbor PageRank, stopping when the token budget is reached.
  Returns node summary + signature, callers/callees/imports/co-change
  neighbors, each with summaries. Compact structured text, not raw code.
- `cgraphy_enrich()` / `cgraphy_store_summaries()` — summarization loop above.

Tool descriptions explicitly tell agents when to call them (e.g., "before
exploring files, call `cgraphy_overview`") — this steering is what produces the
token savings in practice.

## Research-Grade Features (v1)

Three additions that differentiate Cgraphy from grep-style retrieval and are
individually measurable — the seeds of a research paper on graph-native
context management for coding agents:

1. **Importance-ranked retrieval.** After each index, run PageRank over the
   structural edge graph (pure-Python power iteration; no dependency). The
   `rank` score orders `overview` output and re-ranks `search` results, so the
   most load-bearing symbols surface first. Claim: centrality-weighted context
   beats lexical ranking for agent orientation.
2. **Token-budgeted subgraph extraction.** `cgraphy_context` takes a
   `token_budget` and performs greedy best-first graph expansion (priority =
   edge weight × neighbor rank), stopping exactly at budget (tokens estimated
   as chars/4). Cost scales with the question, not the repo — the core claim
   for large-directory work.
3. **Git co-change edges.** `cgraphy index --git-history` mines `git log
   --name-only` for files that repeatedly change in the same commits (logical
   coupling, from mining-software-repositories research) and stores
   `co_changes` edges weighted by co-occurrence frequency (minimum support 3
   shared commits, capped at the last 1000 commits). Static edges say what
   *calls* what; co-change edges say what *breaks* together.

## Research Roadmap (post-v1)

- **Hierarchical community summaries:** Leiden/label-propagation clustering →
  subsystem-level LLM summaries → agents zoom repo → subsystem → symbol
  (GraphRAG-for-code).
- **Usage-feedback reweighting:** log which nodes agents retrieve together in
  a session; strengthen those edges over time — a self-improving retrieval
  graph.
- **Data lineage nodes:** schema-sniff CSV/Parquet/SQL, link code that
  reads/writes each dataset — proper coverage of data directories.
- **Embedding hybrid search:** optional local-embedding vector table fused
  with FTS5.
- **Evaluation harness:** SWE-bench-style tasks answered with vs. without
  Cgraphy; report tokens spent and answer accuracy. Required evidence for any
  paper.

## Viewer (`cgraphy view`)

Python-stdlib HTTP server, one self-contained HTML page (bundled Cytoscape.js
— purpose-built for network graphs; no CDN, no build step). Features, strictly: search box,
color-by-kind, click node → side panel (summary, signature, edges),
double-click → expand neighbors.

## Error Handling

Principle: **never block the agent.**

- Unparseable files get `parse_error` flags and are skipped; indexing always
  completes.
- MCP tools return graceful, informative text instead of raising ("no graph
  found — run `cgraphy index` first").
- Staleness: cheap mtime scan on each MCP call; `cgraphy_search` and
  `cgraphy_context` auto-trigger incremental re-index when stale (self-healing).

## Testing

- pytest; fixture mini-repos per Tier-1 language with golden-file assertions on
  extracted nodes/edges.
- Incremental indexing tests: edit one file → only its nodes/summaries change.
- MCP tools tested in-process with the MCP Python SDK test client.
- An honest token-savings benchmark script: same question answered with and
  without Cgraphy, tokens measured.

## Distribution

- PyPI package, MIT license. Note: the name `cgraphy` may already be taken on
  PyPI; verify at implementation time and fall back to a variant such as
  `cgraphy-mcp` (project name stays Cgraphy).
- README shows one-line setup for Claude Code
  (`claude mcp add cgraphy -- uvx cgraphy serve`), Codex CLI, Gemini CLI, and
  Cursor.

## Decisions Log

| Decision | Choice | Alternatives considered |
| --- | --- | --- |
| Primary goal | LLM context efficiency | Visualization-first; both equally |
| Integration | MCP server | Claude-only skill; VSCode extension; CLI + files |
| Extraction | Hybrid static + LLM summaries | Pure static; pure LLM (GraphRAG-style) |
| Languages | Any language, two-tier (7 Tier-1) | Python-only; Python+TS only |
| Audience | Public open source | Personal tool; commercial |
| Viewer | Minimal built-in HTML page | None in v1; full app |
| Stack | Python all-in-one package | TypeScript/Node; split indexer + reader |
| Storage | SQLite + FTS5 | Kùzu/Neo4j; JSON files in repo |
