\begin{abstract}
AI coding agents spend most of their context re-reading source files to
orient themselves --- a cost paid on every prompt that grows linearly with
repository size. Code knowledge graphs are an increasingly popular remedy,
but recent systems bundle many graph signals at once, leaving open
\emph{which signals actually improve retrieval enough to justify their
tokens}. We present \textbf{cgraphy}, an open-source, language-agnostic
knowledge-graph layer served over the Model Context Protocol (MCP), and use
it as an instrument to answer that question. cgraphy contributes
(1)~importance-weighted, \emph{token-budgeted} subgraph extraction;
(2)~git co-change edges fused with the static graph under leakage control;
(3)~an \emph{agent-self-enrichment} protocol in which the host agent writes
persistent, content-hash-keyed semantic summaries --- no server-side LLM or
API keys; (4)~edit-loop tools (budgeted symbol reading, blast-radius
impact analysis, diff-to-symbol review context) with usage-aware retrieval;
(5)~optional hybrid retrieval fusing FTS with tiny static embeddings by
reciprocal-rank fusion; and (6)~two reproducible, grading-free benchmarks:
a commit-derived localization suite (450 tasks, 9 repositories,
7 languages) and an issue-driven suite built from SWE-bench~Lite instances
evaluated at their pre-fix snapshots. Across 9 repositories, the signal
stack lifts mean hit@10 from 0.602 (lexical baseline) to 0.733, with
semantic fusion the largest single contributor at zero added payload; on
real GitHub issues the full stack gains +15.8 points hit@5 and hit@10
(0.456 $\rightarrow$ 0.614). Gains concentrate where lexical search is
weakest (C++ +28, JavaScript +24 hit@10) --- and on one repository fusion
\emph{hurts}, underscoring that graph signals are measurable options, not
assumed wins. Every configuration answers in a few hundred to
$\sim$1{,}300 tokens; on real private codebases, orientation via the graph
is 134--420$\times$ cheaper than reading the code.\footnote{Code,
benchmarks, and data: \url{https://github.com/pmgarg/cgraphy}}
\end{abstract}

\section{Introduction}
Modern coding agents (Claude Code, Codex, Gemini CLI, Cursor) operate by
listing directories, grepping, and reading files. This \emph{exploration
tax} recurs on every task: the agent reconstructs, from raw text, a mental
model that is almost entirely stable across prompts --- what the modules
are, who calls whom, where the important entry points live. For large
repositories the tax dominates context budgets; for vague prompts
(``fix the login bug'') it compounds, because the agent does not know where
to begin.

A natural remedy is to precompute that mental model as a graph and let the
agent query it. Recent systems validate the idea: RepoGraph~\cite{repograph}
plugs a line-level Python code graph into SWE-bench agents;
CodexGraph~\cite{codexgraph} exposes a code graph database to an agent;
LocAgent~\cite{locagent} frames localization as graph-guided search;
Codebase-Memory~\cite{codebasememory} serves a tree-sitter knowledge graph
over MCP and reports order-of-magnitude token savings; the practitioner
tool Aider popularized PageRank-ranked, token-budgeted repository
maps~\cite{aider}; and classic mining-software-repositories research showed
that files which changed together in the past change together in the
future~\cite{gall,zimmermann}.

These systems bundle signals --- structural edges, importance ranking,
communities, history --- and report aggregate wins. What is missing is a
\emph{decomposition}: which signal earns its place, on which kinds of
codebases, at what token price? The question matters because every token of
graph payload competes with tokens of actual code, and because
Agentless~\cite{agentless} demonstrated that simple retrieval is a
stubbornly strong baseline in this domain.

We build cgraphy to answer it. Design goals are strict: any language
(two-tier tree-sitter extraction), any agent (MCP; one config line each for
Claude Code CLI/VSCode/desktop, Codex, Gemini CLI, Cursor), zero
infrastructure (one SQLite file), zero credentials (the host agent enriches
its own graph), and a hard token budget on every answer.

\textbf{Contributions.}
\begin{enumerate}
  \item \emph{System:} a language-agnostic code knowledge graph with
        PageRank importance, git co-change edges, token-budgeted best-first
        subgraph extraction, and an agent-self-enrichment protocol with
        content-hash summary persistence, served over MCP
        (Section~\ref{sec:system}).
  \item \emph{Benchmarks:} two objective, no-human-grading localization
        suites --- commit-derived (450 tasks, 9 repos, 7 languages) and
        SWE-bench-Lite-derived (real GitHub issues, pre-fix snapshots) ---
        both with explicit leakage control for history-based signals
        (Section~\ref{sec:eval}).
  \item \emph{Findings:} an ablation across six signal levels showing
        (a)~lexical FTS is a strong baseline, (b)~semantic fusion is the
        largest single signal and costs zero extra payload, (c)~graph gains
        concentrate where lexical search is weak (C/C++/JS/Java), including
        one repository where fusion hurts, (d)~co-change helps only where
        history is dense, and (e)~all configurations stay 1--3 orders of
        magnitude cheaper than reading code (Section~\ref{sec:results}).
  \item \emph{Practice:} deployment mechanics that research systems omit:
        agent steering via instruction files, staleness-driven incremental
        re-indexing, and content sniffing that survives real user
        directories (a saved-webpage bundle with 73{,}794 minified JS files
        hung a filename-based indexer for hours; an 8\,KB head-sniff fixed
        it) (Section~\ref{sec:practice}).
\end{enumerate}

\section{Related Work}
\textbf{Repository graphs for agents.} RepoGraph~\cite{repograph} improves
SWE-bench agents with ego-graph retrieval over line-level Python graphs;
CodexGraph~\cite{codexgraph} has agents emit graph-database queries;
LocAgent~\cite{locagent} performs graph-guided localization;
ARISE~\cite{arise} targets fault localization and repair.
Codebase-Memory~\cite{codebasememory} is closest to our system: a
66-language tree-sitter graph over MCP with Louvain communities and
co-change edges, evaluated by author-graded answer quality. cgraphy differs
in mechanism (budgeted rank-weighted extraction; agent-written persistent
summaries; rank-blended search) and, more importantly, in evaluation
philosophy: objective ground truth, signal ablations, and leakage control
rather than graded QA --- Codebase-Memory itself lists systematic
comparison as future work. \textbf{Token-budgeted maps.}
Aider~\cite{aider} fits a PageRank-ranked map into a fixed token budget
inside one CLI tool; cgraphy generalizes the idea to a queryable,
protocol-standard service with per-query budgets. \textbf{Change coupling.}
Logical coupling is classic MSR~\cite{gall,zimmermann}; we measure its
marginal value for agent-facing retrieval with the fix commit excluded from
mining. \textbf{Strong simple baselines.} Agentless~\cite{agentless}
showed pipeline retrieval rivals agentic exploration; our lexical-baseline
finding is consistent and quantified per language.
\textbf{GraphRAG}~\cite{graphrag} popularized hierarchical community
summaries for text corpora; porting them to code is our roadmap.
SWE-bench~\cite{swebench} anchors end-to-end evaluation; we derive an
objective localization suite from its Lite split.

\section{System}\label{sec:system}
\textbf{Indexing.} \texttt{cgraphy index} walks the repository (honoring
\texttt{.gitignore}/\texttt{.cgraphyignore}; skipping binaries, lock files,
and minified assets by content sniff), parses files with tree-sitter, and
writes nodes (\texttt{file}, \texttt{class}, \texttt{function},
\texttt{method}) and edges (\texttt{contains}, \texttt{calls},
\texttt{imports}, \texttt{inherits}) into a single SQLite database with an
FTS5 index over names and summaries. Tier-1 query specifications cover
Python, JavaScript/TypeScript, Java, Go, C, C++, Rust; a generic AST walk
extracts definitions for any other tree-sitter grammar. Re-indexing is
incremental by content hash; cross-file references resolve by qualified
name, best-effort, with unresolved names retained.

\textbf{Ranking.} Weighted PageRank (30 iterations, pure Python) runs over
all non-containment edges after each index; search blends lexical relevance
with importance: $\mathrm{score} = \mathrm{bm25} - 5.0\cdot\mathrm{rank}$.

\textbf{Co-change edges.} \texttt{--git-history} counts file pairs
co-occurring in commits (up to 3{,}000 commits; support $\geq 3$; commits
touching $>50$ files discarded) and adds \texttt{co\_changes} edges with
normalized weights.

\textbf{Budgeted context.} \texttt{cgraphy\_context(symbol, budget)}
expands best-first from the target node with priority
$w\cdot(0.5+\mathit{rank})\cdot 0.6^{\mathrm{depth}}$, emitting compact
node lines until the budget (4 chars/token estimate) is exhausted. Cost
scales with the question, not the repository.

\textbf{Agent self-enrichment.} Semantic summaries are written by the host
agent through a two-tool loop: \texttt{cgraphy\_enrich} returns
unsummarized symbols (importance-first) with snippets;
\texttt{cgraphy\_store\_summaries} persists one-liners keyed by a SHA-256
of the symbol's source, so an edit invalidates only its own summary. The
mechanism needs no server-side LLM, works in every MCP client (unlike MCP
sampling, which few clients implement), and turns enrichment into an
idle-time agent task.

\textbf{Hybrid semantic retrieval (optional).} Static
model2vec embeddings (CPU-only, no torch; \texttt{cgraphy[semantic]})
are computed per node at index time and fused with FTS5 rankings by
reciprocal-rank fusion, which requires no score calibration. This targets
the vocabulary gap between issue-style prose and code identifiers that our
evaluation identifies as lexical search's dominant failure mode.

\textbf{Subsystems and lineage.} Deterministic label propagation over the
file graph yields subsystem communities surfaced in the overview (a
repo$\rightarrow$subsystem$\rightarrow$symbol zoom); dataset files (CSV
headers, SQL \texttt{CREATE TABLE}s) receive deterministic schema
summaries, making data directories searchable alongside code.

\textbf{Edit-loop tools.} Beyond orientation, three tools serve the
change cycle: \texttt{cgraphy\_read} returns one symbol's line-numbered
source under a budget (replacing whole-file reads);
\texttt{cgraphy\_impact} computes the blast radius of a proposed change ---
transitive dependents, tests likely affected, historically co-changed
files; \texttt{cgraphy\_diff\_context} maps the working git diff to the
exact symbols touched, their users, and covering tests. Retrieval is
\emph{usage-aware}: symbols an agent repeatedly focuses on receive a
small, capped priority boost in later expansions (telemetry never leaves
the local database).

\textbf{Serving and steering.} \texttt{cgraphy serve} exposes eight tools
over stdio MCP; \texttt{cgraphy init} installs project configuration
(\texttt{.mcp.json}) plus steering blocks in
\texttt{CLAUDE.md}/\texttt{AGENTS.md} that direct agents to consult the
graph before reading files and to check impact before editing shared code
--- servers cannot force tool use, only steer it. Tools self-heal: a cheap
staleness check triggers incremental re-indexing before answering.
\texttt{cgraphy view} renders the graph locally for humans (Cytoscape.js,
no CDN).

\section{Evaluation Design}\label{sec:eval}
Both benchmarks are objective (no LLM judging, no human grading) and
reproducible from public data.

\textbf{Commit-derived localization.} From each repository's history we
select recent fix-like commits (bug-fix lexicon in the subject; 1--5 code
files changed; files still present at HEAD). The subject is the query; the
touched files are ground truth. Co-change mining \emph{excludes the
evaluated commits}. 50 tasks per repository; 9 repositories spanning
Python, C++, Go, Rust, JavaScript, Java, and C (450 tasks).

\textbf{Issue-driven localization (SWE-bench Lite).} For instances from
seven tractable SWE-bench Lite projects we check out the exact
\texttt{base\_commit}, index that snapshot, query with the real GitHub
issue text (code blocks stripped, first 120 words), and score against the
gold patch's files. Because the snapshot predates the fix, history-based
signals cannot leak by construction.

\textbf{Methods.} An ablation ladder: \textsc{fts} (FTS5 bm25);
\textsc{rank} (+PageRank blending); \textsc{expand-nc} (+one-hop weighted
expansion, co-change removed); \textsc{expand} (full graph).
\textbf{Metrics:} hit@$k$ (first ground-truth file in top $k$), MRR, and
mean retrieval payload in tokens.

\textbf{Commit-derived suite (Table~\ref{tab:loc},
Figures~\ref{fig:hit10}--\ref{fig:gain}).}
Across 450 tasks, mean hit@10 rises monotonically along the ablation
ladder: 0.602 (\textsc{fts}) $\rightarrow$ 0.629 (+PageRank, free)
$\rightarrow$ 0.662 (+expansion/co-change) $\rightarrow$ 0.724 (+semantic
fusion) $\rightarrow$ \textbf{0.733} (full stack) --- a +13.1-point total
gain, with +8.6 hit@5 and +0.076 MRR. Four patterns stand out.
\emph{First}, semantic fusion is the single largest signal and costs
nothing extra: the \textsc{hybrid} rung uses the same payload as
\textsc{fts} (364 tokens mean) yet adds +12.2 points hit@10, with the
biggest jumps exactly where identifiers and prose diverge (junit4 +18
hit@5; express +24 hit@10 over \textsc{fts}; fmt +26 with expansion).
\emph{Second}, gains concentrate where the lexical baseline is weak
(Figure~\ref{fig:gain}): C++'s header/implementation split and JavaScript's
terse naming defeat bm25, and the graph supplies what naming does not.
\emph{Third}, PageRank is free and never harmful on hit@10 (up to +10 on
gin); co-change contributes where history is dense (click +4).
\emph{Fourth}, the signals are not universally additive: on ripgrep,
fusion \emph{hurts} (0.72 $\rightarrow$ 0.66 hit@10) --- Rust's
already-precise identifiers make lexical rank the better prior, and RRF
dilutes it. Signal utility is measurable, repo-dependent, and worth
measuring per codebase --- precisely the argument for shipping the
benchmark with the tool.

\textbf{Issue-driven suite (SWE-bench Lite, Table~\ref{tab:swe}).}
Real issue texts are markedly harder than commit subjects: the lexical
baseline drops to hit@10 0.456 (from 0.602 on the commit suite), because
user-written reports describe symptoms in prose that shares little
vocabulary with code identifiers. This is exactly the regime where
semantic fusion earns its keep: \textsc{hybrid} adds \textbf{+15.8 points
hit@5} (0.333 $\rightarrow$ 0.491) and +12.3 hit@10 at the same payload as
\textsc{fts}, and the full stack reaches \textbf{0.614 hit@10} (+15.8 over
lexical, +12.3 over the best structural rung). PageRank and co-change
contribute little here --- pre-fix histories at often-years-old base
commits offer sparse support. We emphasize the operating point: these are
single-shot, sub-second, LLM-free retrievals costing 450--1{,}300 tokens
that seed an agent's \emph{first} tool call; LLM-in-the-loop localization
agents~\cite{locagent} report higher file accuracy at orders of magnitude
more tokens, and the two are complementary layers, not competitors.

\textbf{Token economy.} Retrieval payloads range 148--1{,}239 tokens per
query across all repositories and methods. On three private codebases
(852, 360, and 592 files; Python and C++), full orientation via
\texttt{overview}+\texttt{search}+\texttt{context} cost 2{,}623--4{,}431
tokens versus 0.59M--1.25M tokens for reading the code:
\textbf{134--420$\times$} reductions, growing with repository size since
the graph payload is budget-capped while code volume is not.

\section{Deployment Practice}\label{sec:practice}
Two findings that research prototypes rarely report. \emph{Steering is
part of the system}: an MCP server cannot force an agent to prefer it;
cgraphy ships instruction-file blocks (\texttt{CLAUDE.md},
\texttt{AGENTS.md}) and imperative tool descriptions, which in our
experience determine whether the graph is consulted at all.
\emph{Real directories are hostile}: a user project containing saved
webpages (73{,}794 JavaScript files of minified code without
\texttt{.min} names) made a filename-based indexer run for hours; an
8\,KB head-sniff for kilobyte-long lines reduced indexing to seconds.
Robustness of this kind is a precondition for any of the retrieval gains
to matter.

\section{Limitations and Threats to Validity}
Commit subjects are terser than real prompts (the SWE-bench suite
mitigates this with genuine issue texts). Ground truth equals the fixing
patch's files --- necessary but not always sufficient context. Nine
repositories and seven languages remain a sample; effects are modest
against a strong lexical baseline and repository-dependent, and we frame
them accordingly. The SWE-bench-derived suite measures localization, not
end-to-end issue resolution; running full SWE-bench with an agent steered
by cgraphy is the next experiment and requires substantial compute.
Token counts use a 4-chars/token estimate. Agent-written summaries were
empty during all benchmarks: embeddings covered only names, qualified
names, and signatures, and expansion used only structural text --- so the
reported numbers are a lower bound on what an enriched graph could
achieve. Usage-aware reweighting is implemented but unevaluated (it
requires longitudinal agent traces); the RRF fusion weights and the
ripgrep regression suggest per-repository signal calibration as concrete
future work.

\section{Conclusion}
cgraphy packages code knowledge graphs as a zero-infrastructure,
any-agent, token-budgeted context layer spanning both the reading loop
(overview, search, context, symbol-level reads) and the editing loop
(impact analysis, diff review), and uses two objective benchmarks to
decompose which signals earn their tokens: semantic fusion (largest,
free at query time, occasionally harmful), PageRank (small, free,
consistent), structural expansion (moderate cost, biggest where lexical
retrieval fails), and co-change (history-quality dependent). The token
economy --- hundreds of tokens per query against 134--420$\times$ more for
reading code --- is the unconditional win. Remaining roadmap: evaluating
usage-feedback reweighting on longitudinal traces, summary-quality
evaluation after agent enrichment, per-repository signal calibration, and
full SWE-bench agent runs.
